# Villie — Application Security Review

**Date:** 2026-07-07
**Reviewer:** Application Security Engineer (one-off AppSec review)
**Scope:** Auth / OAuth, Supabase data layer (RLS), and Edge Functions, emphasizing recently changed surfaces.
**Type:** READ-ONLY review. No code, config, or data was modified.

---

## Summary

Villie's backend is materially more secure than a codebase this size usually is: RLS is enabled on every PII/payment/health table, write paths are almost universally tied to `auth.uid()` or routed through the service role, the Google Sign-In code-exchange keeps client secrets server-side, and the newest money-handling surface (Villie Boxes, migration 092 + `boxes-create-payment-intent` + `stripe-webhook`) recomputes every cent server-side, verifies the Stripe signature over the raw body, and fails closed on a missing secret. The team has also run several deliberate hardening passes (migrations 051–054, 062, 089–090). The most serious finding is not in the new code but in an older, still-live surface: breast-milk **donor home addresses and phone numbers live as plain columns on a table that every authenticated user can `SELECT` in full**, so the "post-payment reveal" RPC that is supposed to gate them can be bypassed with a direct PostgREST query. The other actionable items are a fail-open Calendly webhook, two edge functions (`twilio-sms`, `push-notify`) whose only auth is the platform JWT gate, a non-atomic specialist-invite claim, and a specialist booking charge whose amount is taken from the client. Recent commits in the last 10 days were documentation-only; the last security-relevant change was the Villie Boxes feature (commit `4527820`), which reviewed clean.

**Findings by severity:** Critical 1 · High 3 · Medium 5 · Low 4

## Live confirmation (2026-07-07, read-only DB + config inspection)

Ran after the static review to verify findings against the hosted project:

- **C-1 (donor PII) — CONFIRMED.** `has_column_privilege('authenticated', 'public.milk_donor_profiles', 'address_line'/'phone'/'lat'/'lng'/'social_links', 'SELECT')` all return `true`; the `milk_donor_profiles_select_active` policy (`qual: is_active = true`) is for role `authenticated`. `anon` holds the column grants too but has **no** `anon` SELECT policy, so anon rows are denied — exposure is authenticated-only. **`count(active rows with address) = 0`** → real access-control hole, **zero data currently exposed** (pre-launch). Fix drafted: `migrations/095_security_milk_donor_pii_column_revoke.sql` (+ a small companion `api/milk.ts` change). Not yet applied.
- **H (Calendly webhook) — CONFIRMED.** `calendly-webhook` has `verify_jwt: false` (publicly callable) and fails open on a missing secret. Real; fix = fail closed like `stripe-webhook`/`resend-webhook`.
- **H (twilio-sms / push-notify open relay) — DOWNGRADED.** Both have `verify_jwt: true` at the platform edge, so they are not open relays today. Reclassify as defense-in-depth (add in-code auth so a future `verify_jwt` flip can't silently expose them).
- **Bonus (Supabase advisor):** leaked-password protection is disabled (Pro-tier feature — plan constraint, see CLAUDE.md accepted-warnings), and public buckets `avatars` + `gear-listings` allow listing (already an accepted warning per CLAUDE.md).

**Net after confirmation:** Critical 1 (confirmed, fix drafted) · High 2 actionable (`calendly-webhook`, `specialist-invite-accept` race) + 1 downgraded.

**Most important finding:** Any authenticated user can harvest every active milk donor's home address and phone number with a single direct `milk_donor_profiles` query, because those columns are protected only by an application-layer RPC while RLS grants full-row read (F-1).

## Critical

### C-1 — Milk donor home address + phone are readable by any authenticated user (column exposure vs. RPC gate)
- **What:** `address_line` and `phone` were added directly onto `milk_donor_profiles` (`supabase/migrations/018_v2_m3_purchase.sql:14-16`). That table's SELECT policy `milk_donor_profiles_select_active` (`supabase/migrations/005_v2_milk_rls.sql:5-7`) grants `TO authenticated USING (is_active = TRUE)` — i.e., the **entire row** of every active donor is readable by any signed-in user. Postgres RLS is row-level, not column-level, and there is no `REVOKE ... (address_line, phone)` column grant anywhere in the migrations.
- **Where:** `supabase/migrations/018_v2_m3_purchase.sql:14-16` (columns) · `supabase/migrations/005_v2_milk_rls.sql:5-7` (row policy) · intended gate at `018_v2_m3_purchase.sql:118-135` (`get_transaction_pickup_address`).
- **Why it's exploitable:** The design intends address/phone to be revealed only post-payment through `get_transaction_pickup_address`, which correctly checks `v_recipient = auth.uid()` and `v_status IN ('paid','fulfilled')`. But that RPC is just *one* path to the data. A malicious client with any valid session can bypass it entirely:
  ```js
  supabase.from('milk_donor_profiles')
    .select('display_name, address_line, phone, city, state, zip_code')
    .eq('is_active', true)
  ```
  and dump every active donor's home address + phone with no purchase, no transaction, no authorization check. The app's own API even uses `select('*')` on this table (`apps/mobile/src/api/milk.ts:113-121, 140-152`), so the columns are already flowing. The affected population — breast-milk donors, frequently new mothers with infants at home — makes a home-address leak a physical-safety issue, not just a privacy one.
- **Recommended fix:** Move `address_line`/`phone` off the publicly-readable row. Best option: relocate them to a separate `milk_donor_private` table (1:1, `TO authenticated USING (auth.uid() = user_id)` own-only) and have `get_transaction_pickup_address` join it with the service role. Faster interim mitigation: `REVOKE SELECT (address_line, phone) ON milk_donor_profiles FROM authenticated;` and grant the columns only to the RPC's definer role — but a dedicated table is the durable fix. Then audit `apps/mobile/src/api/milk.ts` to stop `select('*')` on this table.

## High

### H-1 — `calendly-webhook` fails OPEN when the signing secret is unset
- **What:** `verifySignature` returns `true` (accept) if `CALENDLY_WEBHOOK_SECRET` is empty.
- **Where:** `supabase/functions/calendly-webhook/index.ts:16-20`.
- **Why it's exploitable:** If the secret is unset (fresh env, rotation gap, or "local dev" left in place) the function accepts any unsigned request and performs service-role writes to `appointments` from attacker-controlled JSON: it can create `status:'confirmed'` appointments, inject arbitrary `telehealth_link` URLs (later surfaced to users — phishing vector), and cancel other users' appointments by guessing/knowing an `external_id`. This is the opposite posture of `stripe-webhook` (`:68`) and `resend-webhook` (`:64`), which both reject on missing secret.
- **Recommended fix:** Fail closed — `if (!WEBHOOK_SECRET) return false;` (or 500 at startup). Also switch the `sigHeader === expected` compare (`:19`) to a constant-time comparison, matching `resend-webhook`'s `timingSafeEqual`.

### H-2 — `twilio-sms` and `push-notify` have no in-code auth (open relay if the JWT gate ever slips)
- **What:** Both functions accept `{ to/segment, body }` and send an SMS / push with no bearer check, no shared-secret check, and no `auth.getUser()`. Their only protection is Supabase's platform `verify_jwt` default.
- **Where:** `supabase/functions/twilio-sms/index.ts` (whole file) · `supabase/functions/push-notify/index.ts` (whole file). No `supabase/config.toml` pins `verify_jwt=true` per-function, and sibling functions in this repo are documented as deployed with `--no-verify-jwt`.
- **Why it's exploitable:** If either is ever deployed with the JWT gate off (consistent with how neighbors are deployed), it becomes an unauthenticated SMS pump / push-spam relay under Villie's trusted sender identity — toll fraud and smishing. Even with the gate on, the default only checks that *some* valid JWT is present, not that the caller is the service role; any signed-in user's token satisfies it. Legitimate callers already invoke these with the service-role bearer (`milk-safety-screener`, `admin-approve-specialist`, `events-ingest-ics`).
- **Recommended fix:** Add an in-code gate — compare the bearer against `SUPABASE_SERVICE_ROLE_KEY` (constant-time) and reject otherwise, matching `admin-compliance-events`. Belt-and-braces: commit a `config.toml` that pins `verify_jwt=true` for every function that isn't intentionally public.

### H-3 — `specialist-invite-accept` claims the invite non-atomically (token replay / double-accept)
- **What:** The invite is validated with a `SELECT` (`used_at IS NULL`) up front and only marked used with a separate `UPDATE` at the very end, after `auth.admin.createUser` and the `specialists` insert. The "mark used" is best-effort (logs and continues on failure).
- **Where:** `supabase/functions/specialist-invite-accept/index.ts:116-137` (validate) and `:221-234` (mark used, non-fatal).
- **Why it's exploitable:** Two concurrent requests with the same valid token both pass the initial `SELECT` and proceed to create auth users / specialist rows (only an email-collision would 409 the second `createUser`, and that ordering isn't guaranteed). Because the mark-used step can fail without aborting, a live invite can be reused.
- **Recommended fix:** Claim atomically before doing any work:
  ```sql
  UPDATE specialist_invites SET used_at = now()
  WHERE token = $1 AND used_at IS NULL AND revoked_at IS NULL AND expires_at > now()
  RETURNING id;
  ```
  Proceed only if a row was returned; otherwise 409.

## Medium

### F-M1 — `create-payment-intent` (specialist booking) trusts the client-supplied amount
- **What:** The charge `amount_cents` is read from the request body and validated only for `>= 50`; the specialist's real service price is never looked up server-side.
- **Where:** `supabase/functions/create-payment-intent/index.ts:46-53, 62-79`.
- **Why it matters:** A tampered client can set any amount (including the platform-fee base and `transfer_data` destination payout), enabling underpayment. This is exactly the anti-pattern the newer `boxes-create-payment-intent` explicitly guards against by recomputing every cent from a catalog. Medium because V1 booking is a comparatively trusted flow, but it is a real tamper vector.
- **Recommended fix:** Recompute the amount from the specialist/service record server-side; treat the client amount as advisory only.

### F-M2 — `specialist-invite-create` (and two gear moderation fns) trust an unverified JWT `role` claim
- **What:** `isServiceRoleRequest` base64-decodes the JWT payload and trusts `payload.role === 'service_role'` **without verifying the signature**, relying on `verify_jwt:false` + "only real service-role callers can mint this."
- **Where:** `supabase/functions/specialist-invite-create/index.ts:284-299`; same pattern in `gear-moderation-pager/index.ts:71` and `gear-takedown-template-dispatch/index.ts:135`.
- **Why it matters:** The `role` claim in a JWT is attacker-forgeable (just base64) if the signature is never checked. If the gateway is set to `verify_jwt:false` for these functions (as several neighbors are) the admin gate collapses — allowing creation of invites and branded Resend emails to arbitrary addresses. Contrast `admin-compliance-events:94` / `admin-approve-specialist:27`, which compare directly against the service-role key value.
- **Recommended fix:** Verify the JWT signature, or compare a dedicated shared secret (constant-time), rather than trusting an unverified claim.

### F-M3 — `milk-donor-social-links` / donor row: whole-row read exposes more than the app renders
- **What:** Beyond C-1, note that `milk_donor_profiles_select_active` returns the *entire* donor row to any authenticated user for active donors — `social_links` (075), plus any future column added to this table inherits public-to-authenticated read by default.
- **Where:** `supabase/migrations/005_v2_milk_rls.sql:5-7`; columns accreted across `017`, `018`, `075`.
- **Why it matters:** `social_links` is intended public self-attested proof (acceptable, with in-app "not verified" disclaimer). The structural risk is that this table has become a grab-bag where sensitive columns (C-1) sit next to public ones under one permissive row policy. Any new column is public-by-default.
- **Recommended fix:** Split sensitive/private donor fields into an own-only table (see C-1). Treat `milk_donor_profiles` as the "publicly displayable" projection only.

### F-M4 — `milk-shippo-label` buys a real label from unvalidated client dimensions; `milk-dispute-open` persists unvalidated `evidence_urls`
- **What:** `milk-shippo-label` incurs a real Shippo money charge driven by fully client-supplied `from_address`/`to_address`/`parcel` with no server-side bounds on weight/dimensions/country. `milk-dispute-open` writes `evidence_urls ?? null` straight to the DB with no shape/host validation.
- **Where:** `supabase/functions/milk-shippo-label/index.ts:56-113, 142` · `supabase/functions/milk-dispute-open/index.ts:114-127`.
- **Why it matters:** A client can request an oversized/expensive label (idempotency only blocks a *second* label per tx); arbitrary `evidence_urls` strings are persisted and later rendered (stored-content risk). Auth/ownership on both is correct, so these are not IDOR. Capped at medium because the Milk Hub is currently cash-only (dormant paths).
- **Recommended fix:** Validate parcel bounds server-side; constrain `evidence_urls` to an array of your own Supabase Storage hostnames.

### F-M5 — `npi-verify` performs no auth despite its header claim
- **What:** The header says "service role or authenticated specialist claiming own profile," but the code does no auth check; with a `specialist_id` in the body it sets `npi_verified = true` (service-role write) when the NPI resolves as active in the CMS registry.
- **Where:** `supabase/functions/npi-verify/index.ts:61-75, 166-175`.
- **Why it matters:** An unauthenticated caller can flip the verified badge. Impact is bounded by the `.eq('npi_number', npi_number)` guard (the attacker must already know the NPI stored on the row) and by good input validation (`/^\d{10}$/`), so it's medium not high — but a trust badge is exactly the kind of thing that should require caller authorization.
- **Recommended fix:** Require and verify the caller JWT; scope the write to the specialist owning the profile (or restrict to the service role).

## Low

### F-L1 — `reviews` SELECT is readable by `anon`
- **What / where:** `supabase/migrations/002_v1_rls_policies.sql:34` — `reviews_select_all ... FOR SELECT USING (true)` with no `TO` clause, so the policy applies to `public` (includes `anon`). `reviews` holds `user_id` (UUID), `rating`, `body`, `specialist_id`.
- **Why low:** Review bodies are intended public content and the linkage is pseudonymous (UUID, not name/email — `users` is own-row-only). But `anon` exposure is broader than needed; the sibling `specialists` policy was later narrowed to authenticated + approved-only, and `reviews` was missed.
- **Fix:** Add `TO authenticated`.

### F-L2 — Specialist sub-tables (`languages`, `insurances`, `services`, `transportation`) readable by `anon`
- **What / where:** `supabase/migrations/002_v1_rls_policies.sql:27-30` — four `FOR SELECT USING (true)` policies with no `TO` clause. Unlike the parent `specialists` table (gated to approved-only in `013:51`), these sub-rows leak for not-yet-approved specialists too.
- **Why low:** Directory metadata, not user PII.
- **Fix:** Scope `TO authenticated` for consistency with the parent table.

### F-L3 — `SUPABASE_SERVICE_ROLE_KEY` and Twilio SID stored in developer `.env` files
- **What / where:** `apps/mobile/.env:15` holds `SUPABASE_SERVICE_ROLE_KEY`; `supabase/functions/.env:3` holds `TWILIO_ACCOUNT_SID`. Both files are gitignored and **were never committed** (verified via `git log --all` and `git check-ignore`), and the service-role key is **not** `EXPO_PUBLIC_`-prefixed and is not referenced by any `apps/mobile/src` code, so it is not bundled into the shipped app.
- **Why low (not critical):** No repo leak, no client-side leak — the key is used only by admin CLI scripts (`pnpm specialist:invite`). The concern is hygiene: a full-power, RLS-bypassing service-role key sitting in the same `.env` the mobile developer loads is one careless `EXPO_PUBLIC_` rename or screen-share away from exposure.
- **Fix:** Keep the service-role key out of `apps/mobile/.env`; load it from a separate, developer-scoped secret (or a `.env.admin` never opened during app dev). Confirm CI/EAS never echoes it. Rotate if it has ever been shared.

### F-L4 — Non-constant-time secret comparisons on admin/webhook gates
- **What / where:** `admin-compliance-events/index.ts:94` and `admin-approve-specialist/index.ts:27` compare the bearer against the service-role key with `!==`; `calendly-webhook/index.ts:19` compares the HMAC with `===`.
- **Why low:** Remote timing attacks against a 200+ char JWT / HMAC over the network are impractical, but constant-time comparison is the correct, cheap default for secret material — and the repo already has the right helper (`resend-webhook`'s `timingSafeEqual`).
- **Fix:** Route all secret/HMAC comparisons through a constant-time equal.

## Reviewed and clean (notable good patterns)

- **Villie Boxes (newest payment surface):** `boxes-create-payment-intent` recomputes every cent from an embedded catalog with a core-item tamper guard, refuses to hold an unrecorded PI (cancels on order-insert failure), and authenticates the caller via JWT before the service-role write. Migration `092` correctly enables RLS with owner-only SELECT and **no** client write policy. `stripe-webhook` verifies the signature over the raw body (`constructEventAsync`), fails closed on missing secret, and guards `pending_payment`-only transitions.
- **OAuth:** `auth-google-exchange` keeps the Google client secret server-side, returns only the `id_token` (never access/refresh tokens), and treats the one-time auth code as proof — correct for a pre-session sign-in flow. No client secret is bundled: the only `EXPO_PUBLIC_*` "keys" are the anon key (RLS-protected by design), the Sentry DSN, and public Google client IDs.
- **Webhooks done right:** `resend-webhook` (full Svix HMAC over `id.ts.body`, constant-time compare, 5-min replay guard, fail-closed) and `perks-redemption-webhook` (per-network HMAC / IP-allowlist, constant-time, reject reasons not echoed) are model implementations. `gear-boost-activate` verifies the RevenueCat receipt and fails closed.
- **RLS coverage:** Enabled on all 11 milk tables, all 10 community tables, users/reviews/specialists/favorites/appointments, baby tracker (093), daily check-ins (025), and box orders (092). All `USING(TRUE) WITH CHECK(TRUE)` write policies are scoped `TO service_role` (not client-reachable). Write/INSERT/UPDATE policies on PII tables tie to `auth.uid()`. The historical `room_messages` RLS-disabled bug was already fixed with a regression guard (migration 040). `users` has no privilege-bearing columns, so the missing `WITH CHECK` on `users_update_own` is not a privilege-escalation vector.
- **`account-delete`** resolves the caller via JWT (never trusts a body user_id), is idempotent, and correctly scopes out the attorney-gated cascade.
- **Injection/SSRF:** External URL building uses `URLSearchParams`/`encodeURIComponent`; `events-ingest-ics` fetches only admin-registered feed URLs (service-role table), not client input. No SQL built by string concatenation in edge functions. No hardcoded secrets found in tracked code (all `Deno.env.get`).
